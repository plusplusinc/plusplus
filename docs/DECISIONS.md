**2026-07-16 — The durable run record, PR 2: the phone persists the whole-session GPS track as verbatim GPX bytes on `WorkoutSession`, and the run summary + `isOutdoor` flow through the app's interchange mapping (#378)** — The capture/persistence half on the PR 1 contract. **(1) The canonical on-device route blob IS the GPX bytes** (`WorkoutSession.routeData`, `@Attribute(.externalStorage)`): the sync planner re-pulls any path absent from the local file map, so every device must reproduce sidecar bytes byte-for-byte forever — storing the encode-once bytes verbatim makes that trivial, preserves a foreign (hand-dropped/Strava) GPX through sync, and confines writer determinism to one moment per run; `RouteTrack` is parsed on demand for display, never re-encoded. Three denormalized columns (`runDistanceMeters`/`runMovingSeconds`/`runElevationGainMeters`, raw meters/seconds) give cards and the record header cheap truthful stats even when a sidecar never restored. All four are additive optionals with NO property defaults and NO backfill — nil is the truthful value for every pre-GPS and indoor session (the swiftdata.md lightweight-add law; the `= UUID()` shared-constant trap doesn't apply because there are no defaults at all). **(2) Segment banking fixes #348's last-segment-only limitation**: `RunLocationMonitor` banks the live segment into a session-scoped archive on `pause()` and `stop()` (re-base included), so an outdoor→strength→outdoor session records whole; the monitor's `@State`-per-session lifetime IS the archive's reset. `sessionTrack` (segmented, honest gaps) feeds the GPX; `sessionRoute` (flattened) feeds Health, whose single-polyline route can't encode gaps anyway. Accepted consequence, recorded in #378: a mixed run+strength session now classifies as `.running`/`.outdoor` in Health whenever ANY GPS segment exists, not only when a run came last. **(3) Fix hygiene at capture**: a 1 Hz cap (CLLocation burst deliveries would bloat the track and could collide whole-second sidecar timestamps; distance is chord-measured across skipped fixes, not lost) and altitude carried only when `verticalAccuracy` is in 0...25 m, so the elevation series the hysteresis smooths is only readings the receiver vouched for. **(4) Census flips**: `metricsData.isOutdoor` EXCLUDED→EXPORTED on Exercise + SetLog (riding the explicit-profile/carries-profile gates, written only when true); the three run columns EXPORTED via `SessionDTO.run` (both measurements or the field is omitted — a lone one is capture corruption we'd rather drop than validate into history); `routeData` EXPORTED as the repo-layout sidecar, deliberately NOT a bundle field, so its round-trip is the sync layer's to prove (PR 3), not the completeness test's. Known limitations carried from the plan: salvaged (crash-recovered) sessions lose the track (monitor is view-lifetime state), and watch-born sessions still carry no GPX (the watch rebuild #379 owns route sync-back). Unit tests drive `locationManager(_:didUpdateLocations:)` directly (banking, re-base, teleport/accuracy/1 Hz gates, altitude gating — hosted in the app so `allowsBackgroundLocationUpdates` is legal); the completeness test asserts the new fields. ⚠️ Real GPS capture, the pause/resume banking, and the Health classification change need the on-device pass listed in #378. — Verbatim bytes make sync determinism a non-problem by construction, and banking segments at every boundary makes the durable record match what the runner actually did rather than whichever stretch happened to come last.

**2026-07-16 — The durable run record, PR 1: a Kit `RouteTrack` + frozen GPX 1.1 sidecar codec, run summary + `isOutdoor` in the interchange (#378; running push plan settled with Dave)** — Phase 0 of the running roadmap (#378–#383): when a GPS workout finishes today, the track goes to Apple Health and is discarded — nothing durable in the app or the repo. This PR lands the platform contract; the app capture/persistence, sync sidecar wiring, and record UI follow (#378 lists the four-PR shape). **Contract decisions:** (1) **Routes are GPX 1.1 sidecars** — `history/YYYY/<same-basename>.gpx` beside the session JSON, paired by NAMING CONVENTION (`FileLayout.routeSidecarPath`), never a JSON link field: computable from a directory listing, zero byte-churn on session files, numbered `-2` suffixes pair for free. GPX because every running tool reads it (the hackable-data ethos: the raw track is usable outside PlusPlus by construction). (2) **The writer is frozen, hand-emitted strings** (`GPX.encode` — not an XML library; byte identity across macOS/iOS/Linux Foundation is a sync invariant), 6-dp coords / 0.1 m ele / whole-second UTC times, golden byte test; the reader is TOLERANT (XMLParser, accepts foreign files, fractional seconds, rtept; timeless points dropped). `RouteTrack.Fix` QUANTIZES at construction to exactly the writer's precision, so decode∘encode is an exact fixed point (tested). (3) **`SessionDTO.run` carries measurements only** (`distanceMeters`, `movingSeconds`, `elevationGainMeters?`) — splits/pace-curve are deliberately NOT stored: they're pure derivations of the sidecar, and append-only history could never correct a stored derivation if the algorithm improved; `RouteTrack.splits(per:)` ships the algorithm publicly instead. Also additive: `isOutdoor` on `ExerciseDTO` + set snapshots (written only when TRUE, riding the explicit-profile gate; validator flags a bare flag without a distance/pace metric — outdoor means "engage GPS", which needs a metric to feed). Schema stays v1 (everything additive, absent = pre-field meaning). (4) **`InterchangeFiles` now skips non-`.json` paths** — it force-decoded ANYTHING under `history/` as a `SessionDocument`, so the first pulled sidecar would have failed the entire sync pass; this guard had to land BEFORE any writer exists. (5) **Kit derivation choices, all pinned by tests:** haversine on the mean-radius sphere; moving time gated at 0.4 m/s (the `LivePaceMeter` floor, so the finish average agrees with the live readout); split seconds are WALL time within segments (standing at a light is part of what that km took) while gaps between segments never count; elevation gain = centered moving-median (radius 2) + **5 m deadband hysteresis** — 5 because the band must exceed noise PEAK-TO-PEAK: a periodic ±2 m altimeter toggle sails through any median filter (period-2 oscillation IS the window majority; a 3 m band accrued 72 m of fictional climb on a flat synthetic run), and undercounting a true climb by up to one band is the accepted anti-overstate cost; `elapsedSeconds(atMeters:)` is public deliberately — it's the odometer feed #380's segment auto-advance needs. Capture policy (for PR 2): 1 Hz fix cap, no thinning — a 2-hour run ≈ 7,200 points ≈ ~800 KB GPX; route fidelity is the product. Kit 330 / CLI 26 green on Linux; swift-reviewer's one MEDIUM (non-finite lat/ele from a hand-edited sidecar poisoning derivations) fixed by sanitizing in `RouteTrack.init`, plus a Darwin twin of the golden-byte test in PlusPlusTests so the byte freeze is pinned where iPhones actually run. — The record IS the product for a running app, and "your runs as files in your own repo" is the identity move no other running app can copy; the contract lands first so every later layer (capture, sync, UI, watch sync-back) writes against a tested, frozen format.

**2026-07-15 — One forgiving search rule for every search surface: `FuzzySearch` in the Kit; discovery is fuzzy, applying changes never is** — Every user-typed search matched by bare `localizedCaseInsensitiveContains` before this: one typo ("benhc", "dumbell", "sqaut"), a missing space ("pushup", "benchpress"), or a gym abbreviation ("rdl", "ohp") returned nothing. Now ONE Kit rule (`PlusPlusKit/FuzzySearch.swift`, pure, Linux-tested) backs them all: fold case/diacritics, split on non-alphanumerics, and require EVERY query word to land somewhere in the candidate (order-free), where "land" is a scored tier ladder — exact word (1.0) > prefix (0.7–0.9, length-graded) > over-typed inflection ("presses"→Press, leftover capped at 2 chars so "deadlft" can't call Dead Bug home) > word-anchored-subsequence abbreviation ("rdl" walks Romanian→D…l, "ohp" reaches Overhead Press; 2–4 chars only, at 5+ you're spelling words and a misspelling must fall to the typo tier, not outrank its honest match) > mid-word substring (3+ chars) > Damerau-Levenshtein typo rescue against the best candidate-token PREFIX (trailing letters free, so "benhc" is 1 edit from "benchpress" too). Multi-word names also match as one glued token (small demotion) — that's the missing-space class. Symbol-only queries and names ("++", "💪" — a user CAN name a routine "++" here) compare literally on the folded text instead of falling out of the token system (swift-reviewer catch: the blank-query "nothing to narrow by" passthrough must not leak into `bestMatch`/find_items, where a token-free name would have resolved to an ARBITRARY routine — a confidently wrong stat — or listed everything as matches; `bestMatch` additionally resolves a blank query to nothing, never a default winner). **Typo budget is length-gated: 0 edits under 5 letters, 1 through 8, 2 beyond** — at 4 letters one edit made "carl" match the entire Cable family (proved against the real 351-name catalog with a scratch harness; the noisy tiers were tuned there, not guessed). **Scores rank**: `ranked(_:query:key:)` filters + sorts best-first with the incoming order as tie-break, so literal hits always sit above tolerant ones and an alphabetical list stays alphabetical within a tier. Wired everywhere a user searches: `ExerciseFilterState` (exercise catalog + picker; the haystack is now name + muscle-group displayName, so "hamstring curl" finds Leg Curl), `LibraryView.candidateEquipment`, and `RoutineCatalogScreen` (name scored at full weight, the summary/style/exercise-list haystack demoted ×0.75; relevance replaces FEATURED order while searching, explicit Name/Time sorts still win). Operator's read side too: `find_items` fragments are fuzzy and ranked (the cap now keeps the LIKELIEST lines, and the tool's @Guide says typos are okay so the model passes the user's words instead of guessing spellings), and `get_stats` resolves a typo'd exercise/routine scope to ONE canonical history name first, exact-scopes the count to it, and speaks the canonical name — never a silent blend of near-matches, and an unresolvable name still honestly counts zero. **The boundary: ChangeEngine write-target resolution stays exact-or-unique-substring** (the model proposes, the app disposes; a forgiving matcher must never pick which routine gets mutated) — only its "Closest:" miss suggestions ride FuzzySearch now (ranked best-first, with the old prefix/substring/shared-word heuristics kept as fallback for partial overlaps the every-word-must-land rule rejects). Deliberately untouched: name-equality identity checks (import dedupe, library-name uniqueness, seed top-up — identity, not search) and the CLI's `stats --exercise` exact filter (documented behavior; adopting FuzzySearch there is a follow-up if wanted). 16 Kit tests pin the tier semantics; app-side filter/data-service tests cover the wiring (macOS CI). — A search box that punishes a transposed letter reads as broken, and every surface inventing its own leniency would drift; one scored rule in the Kit keeps behavior identical everywhere, keeps ranking honest (forgiving ≠ noisy), and keeps the write path strict where forgiveness would be dangerous.

**2026-07-15 — Rest split from transition: a new round of the same block rests, switching exercises gets a short transition (#369)** — Dave: rest (between sets) and transition (switching stations) are different quantities; the single `restSeconds` was serving both, and inside a superset it was actively wrong — the A→B handoff got the full rest, defeating the point of a superset. **The boundary rule** (both platforms, same logic): after a completed set, the next pending log classifies the pause — same `groupIndex` + *different* `setNumber` = a new round of the same block → REST (block `restSecondsOverride` wins, else the routine default); anything else (the superset partner within a round: same block *same* round; or another block) → TRANSITION. Sessions deliberately do NOT carry it (census: execution pacing config, inert once finished — a record's real gaps live in `completedAt`); `TemplateMerge` needed nothing (field-level by construction). **Watch plan**: additive `PlanRoutine.transitionSeconds: Int?` — a pre-transition phone's plan decodes nil and the wrist rests everywhere, exactly the old behavior. **Surfaces**: routine settings gained a BETWEEN EXERCISES / "Transition" stepper under the rest one; the live label is **SWITCH** (phone rest screen, watch, Live Activity/island via an additive `ContentState.isTransition` flag) — "Transition" names the configured quantity, SWITCH is the live-moment word; `skipRestButton`/`extendRestButton` identifiers unchanged. `Routine.estimatedSeconds` (and the catalog template mirror) now counts (rounds−1) rests per block + transitions at exercise/block boundaries — multi-exercise estimates got shorter and more honest. **Deferred**: per-block transition override; the live-mirror rest op stays kind-agnostic (a watch-initiated pause mirrors onto the phone as REST — cosmetic; the phone materialization of a watch-born session does resolve the routine by name to snapshot its transition for custody switches). **Accepted edges** (swift-reviewer): the pause kind/length freezes at log time, so resizing the pending block DURING a countdown can leave one mislabeled/mislength countdown (self-inflicted, +30s/Skip on screen); and a mid-session one-member resize of a superset makes the lopsided tail's handoffs read as rests (errs long, "round" is genuinely ambiguous once lopsided). Kit 276 / CLI 26 green on Linux; boundary-rule unit tests ride `SessionTests`; ⚠️ the live screens (SWITCH label, 0-transition skip, island label, watch label) need an on-device pass. Closes #369.

**2026-07-12 — Calendar sync identity is the TITLE, not the identifier (#346, Dave on-device: "changed the time and it made a second calendar; disable only removed one, orphaning the first")** — First on-device run of calendar sync (#333) surfaced a duplicate-calendar bug. No Kit change (pure link/plan logic already covers the URL round-trip the event-scan match depends on); app-side EventKit glue, so CI compile-verifies and it needs an on-device re-check (enable → change time → confirm ONE calendar; disable → confirm ALL gone; delete the calendar in Calendar.app → confirm the feature turns off and doesn't resurrect).
